# Story skillars-5.6: Parent Development Portal

Status: done

## Story

As a parent,
I want a dedicated development portal for each of my players that shows aggregated progress from all coaches, with the Skills Radar as the hero and session pack status always visible,
So that I have a single complete picture of my child's development without needing to piece it together from multiple sources.

## Acceptance Criteria

**AC 1: Parent development portal loads with Skills Radar as hero** — Given a parent navigates to `parent/players/:playerId/development`, when the portal loads, then the `SkillsRadarChart` (current composite scores across all coaches) is the hero element rendered above all other content; below the radar: session pack tracker (credits remaining per coach), a narrative summary ("Your weak foot exposure increased 42% this month"), the skill exposure bar chart, neglected skill alerts (if any), and a link to the player timeline; the layout is narrative-first — plain-English summaries precede raw numbers throughout (UX-DR29).

**AC 2: Player switcher causes full portal reload** — Given the parent has multiple player profiles, when they switch between profiles using `ParentChildSwitcher`, then `router.push({ name: 'parent-development', params: { playerId: newId } })` is called, causing the portal to reload for the newly selected player with no data bleed between profiles; the `ParentChildSwitcher` is always visible in the shared header and navigates to the development portal on switch.

**AC 3: Multi-coach aggregation with coach attribution** — Given a parent views a player with sessions from multiple coaches, when the Skills Radar and SLU dashboard render, then data is aggregated across all coaches — no per-coach filter toggle; the coach contribution section shows per-skill attribution: "Marcus contributed 68% of DRI exposure this month" derived from `player_skill_stats` grouped by `coach_id`; coach names sourced via `CoachProfileService.getDisplayNamesByIds(coachIds)` (existing batch method).

**AC 4: Empty state for new player** — Given a parent views the development portal for a player with no completed sessions, when the portal renders, then the radar shows an empty polygon with "No assessments yet" on each node (existing `SkillsRadarChart` behaviour); the SLU chart shows "Sessions will appear here once your first session is completed"; the session pack section shows an "Exhausted / no packs" state with a "Find a coach" CTA linking to `/marketplace` (UX-DR25).

**AC 5: Reports section within portal** — Given a parent views the "Reports" section of the portal, when reports are listed, then all PDF reports generated by any coach for that player appear with coach name, date, and download button; tapping download initiates a direct browser download of the signed PDF URL — no intermediate preview page (reuse `PerformanceReportsPanel.vue` with `isCoach=false`).

**AC 6: Timeline with permanent parent access** — Given a parent views the player timeline from the portal, when the timeline renders, then all event types are visible (sessions, homework, videos, assessments, payments, reviews); parent is never subject to coach access expiry — `TimelineResponse.accessExpired` is always `false` for parents (existing `TimelineQueryService` behaviour per Story 5.5); each event links to relevant detail if accessible (reuse `PlayerTimelinePanel.vue`).

**AC 7: Parent ownership enforced on new endpoint** — Given a parent calls `GET /api/development/players/{playerId}/slu/coach-contributions`, then `@playerOwnershipGuard.check(authentication, #playerId)` is enforced — a parent can only see contributions for their own linked players; ROLE_COACH passes through without ownership check (same pattern as all other development endpoints).

### Backend — Service Layer

- [x] **Task 3: Create `SluContributionService.java`** (AC: 3, 7)
  - [x] File: `src/main/java/com/softropic/skillars/platform/development/service/SluContributionService.java`
  - [x] Inject: `SluRepository sluRepository`, `CoachProfileService coachProfileService`
    ```java
    @Service
    @Transactional(readOnly = true)
    @RequiredArgsConstructor
    public class SluContributionService {

        private final SluRepository sluRepository;
        private final CoachProfileService coachProfileService;

        /**
         * Returns per-skill, per-coach SLU contribution percentages for the last 30 days.
         * Only includes skills where at least one coach has contributed SLU.
         */
        public List<CoachContributionDto> getCoachContributions(Long playerId, Instant since) {
            List<Object[]> rows = sluRepository.findCoachContributionsByPlayerId(playerId, since);

            if (rows.isEmpty()) {
                return List.of();
            }

            // Collect all coach IDs for batch name lookup — avoids N+1
            Set<UUID> coachIds = rows.stream()
                .map(r -> UUID.fromString(r[0].toString()))
                .collect(Collectors.toSet());
            Map<UUID, String> displayNames = coachProfileService.getDisplayNamesByIds(coachIds);

            // Group by skill_code, compute total SLU per skill, then % per coach
            Map<String, BigDecimal> skillTotals = new HashMap<>();
            for (Object[] row : rows) {
                String skillCode = (String) row[1];
                BigDecimal contribution = (BigDecimal) row[2];
                skillTotals.merge(skillCode, contribution, BigDecimal::add);
            }

            List<CoachContributionDto> result = new ArrayList<>();
            for (Object[] row : rows) {
                UUID coachId = UUID.fromString(row[0].toString());
                String skillCode = (String) row[1];
                BigDecimal coachSlu = (BigDecimal) row[2];
                BigDecimal total = skillTotals.get(skillCode);
                BigDecimal pct = total.compareTo(BigDecimal.ZERO) == 0
                    ? BigDecimal.ZERO
                    : coachSlu.multiply(BigDecimal.valueOf(100))
                               .divide(total, 1, RoundingMode.HALF_UP);
                String name = displayNames.getOrDefault(coachId, "Unknown");
                result.add(new CoachContributionDto(name, skillCode, pct));
            }
            return result;
        }
    }
    ```
  - [x] Caller is responsible for computing `since` — e.g. `Instant.now().minus(30, ChronoUnit.DAYS)`. This keeps the service testable with arbitrary windows and consistent with future configurable periods.
  - [x] Imports needed: `java.math.BigDecimal`, `java.math.RoundingMode`, `java.time.Instant`, `java.util.*`, `java.util.stream.*`.
  - [x] `displayNames.getOrDefault(coachId, "Unknown")` — guard for coaches whose profile was deleted.
  - [x] Do NOT filter by minimum percentage threshold — return all coaches with any contribution.

    const props = defineProps({ contributions: { type: Array, default: () => [] } })

    // Only top contributor per skill, min 30% threshold to reduce noise

      const bySkill = {}
      for (const c of props.contributions) {

## Dev Notes

### No Flyway Migration Needed

Story 5.6 is pure read-only aggregation and frontend work. ### `player_id` is BIGINT — Consistent with Stories 5.1–5.5

All `development.*` tables use `BIGINT` for `player_id` (TSID `Long`). This is confirmed in `PlayerSkillStat.java:24`. The new native query uses `:playerId` as `Long`.

### Coach Contribution Percentage Precision

`SluContributionService.getCoachContributions` computes percentage as `coachSlu.multiply(100).divide(total, 1, HALF_UP)` — multiply **before** dividing to preserve precision. The prior pattern of `divide(total, 2).multiply(100)` truncated to 2dp first, producing 33.0% for a one-third share instead of 33.3%. Percentages for all coaches on the same skill sum to 100 ±0.2 (1dp rounding across N entries). IT assertions must use a delta, not an exact equality check.

In `CoachContributionSection.vue`, filter to top contributor per skill with ≥ 30% threshold — avoids cluttering the narrative for skills with many coaches sharing small fractions.

### `CoachProfileService.getDisplayNamesByIds` — Verified Batch Method

`CoachProfileService.getDisplayNamesByIds(Set<UUID> coachIds)` exists at line 362 and returns `Map<UUID, String>` (display names). Use this for O(1) batch lookup in `SluContributionService` — never call `getPublicProfile(coachId)` in a loop (N+1 anti-pattern). This is the same pattern established by `ReportGenerationService.listReports` in Story 5.5.

### `SkillsRadarChart.vue` — Parent Read-Only Mode

Pass `readonly=true` and empty `selectedSkillCodes=[]` so all skills with composite scores appear in the radar (no filter) and no coach-preference write occurs. This is consistent with the existing dashboard's `isParent` path (line 29 of `PlayerDevelopmentDashboardPage.vue`: `:readonly="!isCoach"`).

### `PerformanceReportsPanel` — `isCoach=false` Hides Generate Button

From Story 5.5 implementation: `PerformanceReportsPanel.vue` shows the "Generate Report" button only when `isCoach=true && !isScoutTier`. Passing `isCoach=false` from the parent portal ensures the button is absent (AC5 — parents view only, no generate action). The reports list is always shown.

### Session Pack Tracker — `bookingStore.sessionPacks` Data Shape

`bookingStore.sessionPacks` (set by `loadPlayerPacks(playerId)`) returns `SessionPackPurchasedResponse` objects with `{ id, coachId, coachDisplayName, sessionCount, creditsRemaining, status, expiresAt, ... }`. All packs (ACTIVE, EXHAUSTED, EXPIRED, PAUSED) are returned and shown in the list so the parent can see full history. The "Find a coach" CTA (AC4) must appear whenever no ACTIVE pack exists — not only when the list is empty. The page uses `hasActivePacks = sessionPacks.some(p => p.status === 'ACTIVE')` to gate the CTA independently of list length.

### `ParentChildSwitcher.vue` — Always Navigates to Development Portal

After this story, switching players ALWAYS navigates to `parent-development` for the new player. This is intentional per AC2 — the development portal is the canonical parent home. The existing `ParentPlayerPortalPage.vue` (sessions page) retains its own `watch(() => playerStore.activePlayerId)` for backward compatibility when visited directly via bookmark; that page now also handles the case where it reloads data for the new player (no change needed there).

### `CoachContributionSection.vue` — First Name Extraction

`firstNameOf(displayName)` takes the first space-delimited word of the coach's display name (e.g. "Marcus Alves" → "Marcus"). If the display name is a single word (e.g. "Marcus"), it returns the full name. If null/undefined (coach profile deleted), falls back to `displayName ?? ''`. This is a purely cosmetic frontend transform — the backend always sends the full `coachDisplayName`.

### Page Router Watch vs ParentChildSwitcher Navigation

Both the `watch(() => playerStore.activePlayerId)` in `ParentDevelopmentPortalPage.vue` AND the `router.push` in `ParentChildSwitcher.vue` trigger on player switch. They both navigate to `parent-development`. Vue Router deduplicates same-route navigations, so there is no double-navigation. The watch short-circuits when `newId === playerId.value` (same player selected).

### Integration Test — Seeding `player_skill_stats` via JdbcTemplate

`player_skill_stats` is immutable — no JPA delete/save path for test data. In the IT, seed rows directly via `jdbcTemplate.update(...)` and clean up via `jdbcTemplate.update("DELETE FROM development.player_skill_stats WHERE player_id = ?", PLAYER_ID)` in `@AfterEach`. Do NOT use `SluRepository.save()` — it exists but no `delete()` is available per the "IMMUTABLE: append-only" constraint. This is the same pattern as SLU data seeding in Stories 5.2–5.5 ITs.

### `SluContributionService` — Coach Profile Missing Guard

If a coach's profile no longer exists (deleted account), `getDisplayNamesByIds` will omit that coach's entry from the returned `Map`. The `displayNames.getOrDefault(coachId, "Unknown")` guard in `SluContributionService` prevents a NPE/missing-key exception and shows "Unknown" in the narrative. This is best-effort — a deleted-coach contribution still shows with "Unknown" as the name rather than silently disappearing from totals.

### Tab Navigation (UX-DR20) — Deferred

Epic dev notes reference "bottom tab navigation under Parent role (UX-DR20)" for the parent portal. This story does not implement tab navigation — the portal is a standalone `q-page`. If UX-DR20 mandates a tab shell, that must be introduced in a follow-on story or as part of the parent layout route configuration. Do not block this story on it.

### Endpoint Placement — Architecture Note

The new `GET /slu/coach-contributions` endpoint is added to `SkillExposureResource` for pragmatic scope. The epic spec named `ParentDevelopmentPortalResource` as a dedicated controller. If a composite-DTO portal endpoint is added in a future story, it should live in a new `ParentDevelopmentPortalResource` class — do not continue adding parent-portal concerns to `SkillExposureResource` beyond this endpoint.

The existing `player/development/:playerId` route (`PlayerDevelopmentDashboardPage.vue`) serves **both coaches and parents** and is accessed from the coach's command center. The new `parent/players/:playerId/development` portal is parent-only and is the "home" that `ParentChildSwitcher` always navigates to.
