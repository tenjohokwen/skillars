# Decision Log — Video Storage & Streaming Module PRD

_Canonical memory and audit trail for this PRD run. Every decision, change, and override is recorded here._

---

## Session: 2026-05-29

### DL-1: QuotaProvider interface pattern

**Decision:** Module provides a `QuotaProvider` interface (option a) that the consuming app implements. Module calls it before creating an Upload Session.

**Alternatives considered:**
- (b) Module ships configurable quota defaults via Spring properties — rejected; quota limits are business-specific values that differ between apps.
- (c) Quota out of scope entirely — rejected; the module orchestrates the upload lifecycle (reservation, commit, release) and must hook quota enforcement into that sequence to prevent race conditions.

**Rationale:** Keeps quota enforcement integrated into the module's lifecycle orchestration while leaving all limit values and enforcement logic to the consuming app.

**Status:** Captured in FR-29, FR-30.

---

### DL-2: ownerId replaces teamId in Video data model

**Decision:** The Video entity carries `ownerId` only. No `teamId` field. Team or group associations are the consuming app's concern.

**Rationale:** "Team" is a domain concept that differs between apps (coaching teams ≠ social groups). The module must not encode app-specific relational semantics in its data model.

**Status:** Captured in §3 Glossary (Owner), §4.7 (Visibility), data model.

---

### DL-3: Moderation out of scope

**Decision:** Moderation (DISABLED/PRE_PUBLISH/POST_PUBLISH modes) is not a module feature in v1.

**Rationale:** Moderation policy is a business-level decision that varies between a social app and a coaching platform. The module's Access State `BLOCKED` serves as the hook if a consuming app needs to act on moderation outcomes, without the module dictating policy.

**Status:** Captured in §6 Non-Goals.

---

### DL-4: Service/repository layer + optional admin REST endpoint

**Decision:** The module exposes a service/repository layer only. Consuming apps build their own REST controllers for end-user video operations. The module may expose a base admin REST endpoint as an extension point.

**Rationale:** Shipping end-user REST controllers would constrain the consuming app's API design and conflict with the template's goal of being composable without source edits.

**Status:** Captured in FR-31, FR-32, §6 Non-Goals.

---

### DL-5: TEAM visibility mode retained as a stored label

**Decision:** Despite removing `teamId` from the data model, the TEAM Visibility mode is retained as a stored attribute. Enforcement of TEAM semantics is delegated entirely to the consuming app.

**Rationale:** Removing TEAM from Visibility would make the module less expressive for apps that need group-scoped video access. Storing the label without enforcing it keeps the module generic.

**Open question:** Whether `TEAM` should be renamed to a more neutral term (e.g., `GROUP`) — logged as OQ-1.

**Status:** Captured in FR-33, FR-34, OQ-1.

---

### DL-6: No-op QuotaProvider default

**Decision:** The module ships a no-op `QuotaProvider` default that approves all quota checks. Apps that need enforcement supply their own implementation.

**Open question:** Fail-fast at startup vs. silent default — logged as OQ-3.

**Status:** Captured in FR-29, OQ-3.

---

---

### DL-7: TEAM visibility renamed to GROUP

**Decision:** Visibility mode `TEAM` renamed to `GROUP` throughout the module API and data model.

**Rationale:** `TEAM` carries coaching-platform semantics. `GROUP` is neutral and fits social apps, workspace tools, and coaching platforms equally.

**Status:** Applied to §3 Glossary, FR-33, data model. Counter-metric SM-C1 updated accordingly.

---

### DL-8: Admin REST endpoint included in v1

**Decision:** The admin REST endpoint (FR-32) is included in MVP scope.

**Rationale:** Consuming apps need an extension point for admin operations from day one; deferring it forces every app to build from scratch rather than extending the module's base.

**Status:** FR-32 changed from MAY to MUST. §7.1 updated.

---

### DL-9: QuotaProvider fail-fast at startup

**Decision:** The module fails fast at application startup if no `QuotaProvider` bean is registered. `NoOpQuotaProvider` is available as an explicit opt-in, not a silent default.

**Rationale:** Silent defaults in security-adjacent flows (quota enforcement) create production incidents when developers forget to wire an implementation. Fail-fast surfaces the gap at startup, not in production.

**Status:** Captured in FR-29. Assumptions Index updated.

---

### DL-10: PlaybackToken persisted as database records

**Decision:** Playback Tokens are persisted to the database. Revocation (FR-18) is implemented via `revokedAt` column; server-side validation checks this field on every playback authorization request.

**Rationale:** Persisted records make revocation clean and explicit — no blocklist TTL race, no stale JWT window. The write overhead per token issuance is acceptable given the playback token generation SLA of < 200ms.

**Status:** FR-18 note updated. Data model in addendum already includes `revokedAt`.
